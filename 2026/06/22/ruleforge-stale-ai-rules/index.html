代码过期会让 CI 变红，可你给 AI 写的规则文件过期了，什么都不会发生

发表于2026-06-22|更新于2026-06-22|技术
|总字数:1.6k|阅读时长:5分钟|浏览量:

现在用 Claude Code、Cursor、Copilot 的人，多少都写过一份给 AI 看的规则文件，CLAUDE.md、.cursorrules、copilot-instructions.md 之类。给 AI 一点项目上下文，它干活确实靠谱很多。

但这类文件有一个很隐蔽的毛病：它会过期，而且过期的时候不报错。

代码过期了，测试会挂、CI 会变红、type checker 会叫。规则文件不会。你三个月前写「用 yarn 装依赖、用 black 格式化」，后来项目悄悄迁到了 pnpm、换成了 ruff，那份规则文件还安安静静躺在仓库根目录，一脸笃定地告诉 AI 用 yarn 和 black。AI 信了它，于是它建议的命令跑不通、它格式化出来的代码跟项目风格对不上，你还纳闷这助手怎么越用越笨。

更常见的是另外两种情况。一种是压根没写规则文件，把 AI 本可以拿到的项目上下文白白浪费掉；另一种是从网上抄了一份通用模板，跟你真实的技术栈根本不匹配，通篇「请遵循最佳实践」这种正确的废话。

RuleForge 想解决的就是这一摊事。它做两件事，生成，和体检。

生成：从项目配置里读，不靠猜

生成这部分比较直观。它读你项目的真实配置，pyproject.toml、package.json、lockfile、lint 配置、CI 文件，识别出语言、框架、包管理器、lint 和格式化工具、测试框架、CI 系统，然后生成贴合这套技术栈的规则文件。一个命令支持六种格式：CLAUDE.md、.cursorrules、copilot-instructions.md、工具无关的 AGENTS.md、.windsurfrules、.clinerules。

📸 建议此处插入截图：ruleforge scan . 的输出，列出检测到的语言/框架/包管理器/lint/CI，直观展示「它真的读懂了我的项目」。

从 CI 里提取真实命令

生成里有个值得单独讲的细节：它会去读 GitHub Actions 工作流的 run 步骤，把 CI 真正执行的验证命令提取出来写进规则。

为什么这点重要？因为你「以为」项目怎么测，和 CI「实际」怎么测，经常是两回事。随着时间推移，本地开发习惯、README 里的说明、和 CI 里实际跑的命令，会渐渐漂移出去。一份只反映你记忆的规则文件，会让 AI 给出跟 CI 结果不一致的指引，最后还是只能靠 push → CI 报错 → 来回改的方式磨。

RuleForge 直接把 CI 的 run 步骤提出来，引用了 ${{ secrets.* }} 的命令会被跳过，不会把需要凭据的命令暴露进规则文件。

audit：规则够不够全

光会生成不够，真实世界里满地都是已经写好、但早就不可信的规则文件。这时候就要靠体检。

audit 回答「这份规则文件够不够」。它按几个维度打分：项目背景和技术栈是否覆盖、测试/构建命令是否有明确指引、CI 提取的真实验证命令是否写进去、有没有 secret 处理规则、有没有交代 git 和 review 工作流、有没有对 AI 行为方式的约束。

--format sarif 这条比较实用：把缺失的规则维度直接转成 Code Scanning 告警，合并进 GitHub 的 Security 面板，不需要单独维护一套告警入口。

lint：揪出正在骗 AI 的内容

lint 回答另一个更尖锐的问题：这份规则里有没有写错的、会把 AI 带偏的指引。它专门揪三类：

没填完的占位符：TODO、{{ your project name }}、<tech stack>，模板抄来没改完，AI 会当成真实指令照做。

自相矛盾的工具指令：同时叫 AI 用 npm 和 pnpm、pytest 和 unittest、black 和 ruff。这类冲突在两个人分别更新规则文件时特别容易出现，而且出了问题很难排查原因。

过时的建议：仓库明明已经是 pnpm-lock.yaml 了还让用 yarn，项目早切到 ruff 了还让用 black 格式化。

📸 建议此处插入截图：ruleforge lint . 揪出一份旧 CLAUDE.md 里的占位符或「项目已用 pnpm 却让 AI 用 yarn」的过时建议，这是最有冲击力的一张图。

冲突和过时检查只在同一生态内比较——一个真正同时用 pytest 和 jest（Python + Node 混合）、或 ruff 和 eslint（Python formatter + JS linter）的多语言仓库，不会被误报成「冲突」。

一个关于克制的设计决定

生成新规则时，RuleForge 会先检测项目里已有的助手规则文件，并在生成稿里标出哪些地方被已有的本地规则覆盖，避免无意中覆盖掉你手写的、可能更严格的本地约束。这是一个「帮倒忙代价太高」的判断：帮你生成一份差点的规则，比替你清掉一份你精心维护的规则，风险要小得多。

边界也写进了 README：检测基于配置文件和扩展名，不做代码语义分析；生成的是一个好起点，不是终稿，按项目实际约定再 review 一遍比较稳。它不试图替你写出完美的规则，它试图保证你那份规则没在骗 AI。

而「规则文件在悄悄骗 AI」这件事，恰恰比「生成」本身更容易被忽略，也更值钱。

项目地址：github.com/he-yufeng/RuleForge，pip install ruleforge 就能用，MIT。
