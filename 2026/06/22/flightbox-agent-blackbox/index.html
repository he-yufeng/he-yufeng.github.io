整个技术栈里，只有 LLM 调用这一层不能回放

发表于2026-06-22|更新于2026-06-22|技术
|总字数:1.9k|阅读时长:6分钟|浏览量:

做 Agent 做久了，一定会撞上这样一个晚上。

线上跑得好好的系统，某个用户来反馈：「它今天给我的回答很奇怪。」你打开日志，想把那次出问题的运行复现出来，于是把同样的输入再喂一遍。结果这次它表现得完全正常。你不死心，再跑一遍，还是正常。那次真正出问题的运行，像从来没发生过一样，你怎么都抓不回来。

这种 bug 特别折磨人，因为它跟我们熟悉的那种不一样。传统软件的 bug，你有堆栈、有复现步骤，照着走一遍它就老老实实出现。Agent 的 bug 经常是「一次性」的：发生过，然后消失，只在某个用户的截图里留了个影子。

原因其实就三条，但每一条都致命。第一，LLM 本身是非确定性的，同样的输入，下一次的输出不保证一样。第二，你没法重放当时那一整串调用序列，agent 那一轮跑了几步、每步模型返回了什么、触发了哪些工具，全过去了。第三，你的日志里大概率只有最后那条输出，中间每一步模型到底回了什么、工具收到的参数是什么，根本没留下来。

我后来想明白一件事：我们整个技术栈，几乎每一层都是可回放的，唯独 LLM 调用这一层，我们把它留成了不可回放的。

数据库挂了，你有快照可以恢复到出事那一刻。一个 HTTP 请求出问题，你把那条 curl 原样再发一次就复现了。连分布式系统这种最难缠的东西，业界都在用确定性模拟去重放同一串事件。我们对「可复现」这件事是有执念的，也有一整套工具。可偏偏到了 agent 这儿，那个最不确定、最容易出妖蛾子的环节，也就是 LLM 调用，我们却任由它非确定、不留痕、过完就忘。

飞机为什么要装黑匣子？恰恰因为空难这种事，罕见、严重、而且几乎不可能在事后复现。你不可能让飞机再炸一次给你看。所以你得在它正常飞的时候就把一切录下来，出事之后靠录像倒推。

Agent 的 bug 是同一种形状的问题。所以我写了 FlightBox，给 AI Agent 装一个这样的黑匣子。

录制：一个 with 就够了

它的录制，简单到几乎不像在加东西。你用一个 with 把 agent 代码包起来就行，块里面的调用一个都不会漏，业务代码一行都不用动：

这一块里每一次 chat.completions.create()，连同完整的请求、响应、延迟、token 用量、工具调用，都被写进一个本地 SQLite。纯本地，不走任何云，不上报任何地方。

回放的时候，被接管过的方法不再真的去打 API，而是把当初录下来的那条响应原样还给你。于是你的 agent 在本地变成了完全确定性的。你可以在那次「奇怪的回答」上反复打断点、加日志、改逻辑，每一次都精确重演同一条轨迹，直到把根因揪出来为止。这跟「再跑几遍碰碰运气」是两个世界的事。

为什么是 monkey-patch SDK 这一层

这部分我觉得最值得说。FlightBox 是去接管 OpenAI 和 Anthropic 这两个官方 SDK 的方法（chat.completions.create 和 messages.create），而不是去拦更底层的 HTTP，也不是去耦合某个框架的内部。为什么挑这一层？

因为 SDK 的这个方法，是所有上层框架最终都会收敛到的那个最窄、最稳定的缝。你用 LangChain 也好、CrewAI 也好、Pydantic AI 也好，绕来绕去，最后都会落到 openai.chat.completions.create 这一下。补住这一道缝，你就一次性接住了所有框架，还不用去理解任何一个框架的内部实现，也不会因为它们升级而碎掉。选对要拦的那一层，比拦本身更重要。

数据流其实很直白。每次调用进来，FlightBox 先看当前是不是处在 replay 上下文：是的话就拿 run_id 去本地 SQLite 查出对应那一步的响应直接返回，整个过程不碰网络；不是的话就照常真打 API，把请求、响应、延迟、token、工具调用一并落库。一条录制大致就是这样一张表：run_id、step（第几步）、request_json、response_json、latency_ms、tokens、tool_calls。结构简单到你可以直接拿 sqlite3 打开看，这也是我刻意要的，黑匣子的数据不该是黑盒。

一次真实的、尤其是出过问题的生产运行，直接变成一个 pytest 回放测试。这一步的意义在于，你抓到的那个 bug，从此变成了一条回归测试。以后你改代码，再也别想让它悄悄复活。再往大里想一层：你线上每一次有价值的真实流量，都能沉淀成测试用例，你的回归测试集是从真实世界里长出来的，而不是你拍脑袋编的。也可以 -f jsonl 导成评测数据集。

边界要说清楚

免得期待错位。FlightBox 拦的是经过 OpenAI/Anthropic SDK 的调用这一层，它录的是调用层的真相。你 agent 内部那些非 LLM 的随机性，比如你自己代码里的 random、当前时间、某个外部接口的返回，不在它的录制范围里。它解决的是「LLM 调用不可复现」这个最大的不确定性来源，不负责消灭你代码里所有的不确定。把这条记住，它就是个非常趁手的东西。

顺一句，如果你也在做 agent 的测试，它能和我另一个项目 AgentProbe（一个给 agent 行为做断言的 pytest 插件）串起来用：FlightBox 负责把真实运行录下来、变成确定性的回放，AgentProbe 负责在这条确定的轨迹上断言它该调几次工具、该不该走某条分支。一个管录下真相，一个管在真相上立规矩。

项目地址：github.com/he-yufeng/FlightBox，pip install flightbox 就能用，MIT。如果它说中了你某个调试到崩溃的夜晚，去仓库点个 star，或者提个 issue 告诉我你最想录下来的是哪一类调用。
