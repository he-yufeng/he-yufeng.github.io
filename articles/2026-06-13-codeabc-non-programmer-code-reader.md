# 不会写代码，也可以读懂代码：CodeABC 的产品边界

越来越多不写代码的人，也开始被迫和代码打交道。

研究生要改导师留下的 Python 脚本。产品经理要确认外包到底交付了什么。创业者拿到一份代码仓库，想判断它是不是能跑、有没有明显问题。数据分析师只是想把一个参数从 A 改成 B，却要先面对几十个文件和一堆陌生术语。

把代码贴给 ChatGPT，让它解释一下，当然有用。但非程序员面对的通常不是“这一段代码什么意思”，而是一个更大的问题：这个项目从哪里开始看？哪些文件重要？哪些是构建产物？如果只是想改一个配置，大概该找哪里？

这和程序员的需求不一样。

Cursor、VS Code、Copilot 默认用户已经会写代码。它们的入口是编辑器，核心动作是补全、跳转、重构、运行、调试。对只想读懂代码的人来说，信息密度太高，默认界面也太工程师视角。

所以我做了 [CodeABC](https://github.com/he-yufeng/CodeABC)，一个给非程序员用的 AI 代码阅读器。

它的目标很简单：不用先学完整编程，也能开始读懂一份代码。

## 它先回答项目级问题

CodeABC 支持两种入口：拖入本地项目文件夹，或者粘贴 GitHub 仓库链接。它会扫描项目，过滤掉 `node_modules`、`dist`、构建产物、压缩 bundle、缓存、生成文件这类不该让普通人第一眼看到的噪声，然后生成一份大白话的项目说明书。

这份说明书主要回答几个问题：

- 这个项目到底是干什么的？
- 哪些文件重要，哪些可以先不用看？
- 如果只是想跑起来，需要哪几步？
- 如果只是想改一个配置，大概该看哪个文件？

这里最关键的不是“把所有文件都丢给模型”。真实仓库里有大量噪声，模型会浪费上下文，用户也会被无关文件吓退。CodeABC 的 scanner 会先做过滤，再限制文件数量和大小，只把更像人写的源代码的部分送进分析流程。

GitHub 链接入口使用 shallow clone，不拉完整历史。它不是一个无限大的代码理解系统，而是把第一遍阅读控制在可解释、可等待、可缓存的范围内。

这里我做了一个很重要的取舍：CodeABC 不把“尽可能多塞上下文”当成目标，而是把“让非程序员第一眼看到正确层级的信息”当成目标。

普通人读代码时最容易被三类东西误导。第一类是依赖和构建产物，看起来文件很多，其实不是项目逻辑。第二类是框架样板，比如入口文件、路由注册、自动生成的配置，重要但不应该从这里陷进去。第三类是真正承载业务行为的文件，比如数据处理、规则判断、API handler、模型调用、配置读取。

CodeABC 的扫描流程会先把第一类尽量排除掉，再把第二类标成“项目骨架”，最后把第三类作为主要阅读对象。这样生成出来的说明书不是简单文件列表，而更接近“这个仓库的阅读路线”。

我还刻意没有把它做成问答机器人。问答入口看起来最自然，但对非程序员来说，第一句就不知道该问什么。更好的顺序是先给一张地图：项目做什么、入口在哪里、核心文件是什么、运行步骤是什么。等用户有了第一层理解，再让他按文件和代码行继续追问。

## 逐行批注比整段解释更适合阅读

CodeABC 里最有用的是代码视图。

打开一个文件后，不是直接丢给你几百行代码。你可以把鼠标悬停在代码行上，看到 1–3 行粒度的中文批注。它会尽量用普通人能理解的话解释：这几行在做什么，为什么这里有这个数字，为什么这里要等待，为什么这里要循环。

整段解释听起来可能正确，但用户仍然不知道哪几行对应哪个行为。逐行悬停批注的好处是：用户可以把代码当成一篇带注释的文章读。

想粗略看，就读项目说明书。想定位细节，就点进文件。看到不懂的行，再看批注。

## 技术实现刻意保持轻量

后端是 FastAPI + uvicorn，LLM 层用 litellm，可以接 OpenAI、Claude、DeepSeek、Kimi 等兼容模型。分析结果和项目内容缓存在本地 SQLite 里，重复打开同一个项目不用每次都重新等模型生成。

前端是 React + Vite + TailwindCSS，代码高亮用 Shiki，悬停批注用轻量 tooltip 交互，而不是把用户带进完整 IDE。

CodeABC 也保留 BYOK 模式。用户可以在浏览器里填自己的 API key，key 只保存在本地浏览器 localStorage，不需要交给我的服务端。这个设计对这类工具很重要：很多人看的代码不是公开 demo，而是课程作业、公司外包、内部脚本或自己的项目。

## 它不是什么

CodeABC 不是编程学习平台，也不是完整 code review。

它不会保证业务逻辑完全正确，不会替你审安全漏洞，也不会替代资深工程师的判断。它更像第一遍翻译和地图：先把代码从一堵墙变成可以阅读的结构。

等你知道项目大概长什么样，再决定要不要找工程师深挖、要不要自己修改、要不要继续学习。

我觉得这类工具会越来越有用。AI 让生成代码变容易以后，会有更多非程序员拿到代码、修改代码、审阅代码、运行代码。下一步的需求不只是“帮我写代码”，还会是“帮我理解这堆代码到底意味着什么”。
