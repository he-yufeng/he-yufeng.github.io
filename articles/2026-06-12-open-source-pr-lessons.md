# 我复盘了一百多条开源 PR：真正容易被 merge 的修复长什么样

这段时间我给不少开源项目修过 bug，包括 LLM serving、Agent 框架、MCP、RAG、AI observability、推荐系统和一些 coding agent 项目。

一个很直接的感受是：开源 PR 不是“改了代码就完事”。尤其是现在 AI 写代码变快以后，低质量 PR 也变得更多。维护者真正缺的不是更多 diff，而是更少噪声、更清楚的证据、更容易 review 的修复。

真正容易被 merge 的 PR，通常有几个共同点。

先说一个判断标准：好 PR 不是“看起来很聪明”，而是 reviewer 能在很短时间内回答三个问题：

- 这个 bug 现在真的存在吗？
- 这个 diff 为什么正好改在这里？
- 我怎么知道它没有把问题挪到别处？

很多失败的 PR 卡住，不是因为代码完全不对，而是因为这三个问题回答得不够清楚。维护者没有上下文时，只能把它当成风险处理。

## 问题必须真实存在

一个 issue 看起来像 bug，不代表当前 main 上还存在。一个错误日志看起来像根因，不代表真正的失败路径在那里。很多仓库更新很快，issue 里的问题可能已经被另一个 PR 修掉，也可能只在旧版本存在。

动手前我会先查同主题 open PR，查 closed / merged PR，尽量在当前 main 或接近 main 的环境复现。哪怕不能完整跑全量环境，也要给出足够小的证据链：哪条路径会走到这里，哪个输入会触发，为什么不是已有逻辑覆盖。

## 修复要打在根因上

最容易被关的 PR 往往是“看起来修了报错”，但其实只是绕开症状。比如某个字段为空就加一个默认值，某个异常就 catch 掉，某个测试失败就改断言。这样的 patch 有时能让当前 case 过，但维护者会担心它掩盖了更深的问题。

更好的做法是把失败路径拆清楚：状态在哪里变坏，协议约束在哪里被破坏，缓存什么时候失效，配置优先级在哪里被覆盖，异步流程在哪里丢了上下文。修复应该让这个约束重新成立，而不是只让报错消失。

比如 LLM serving 里一个 cache 状态没有在 recompute 路径重置，表面现象可能只是后续请求拿到异常状态。低质量修法是看到异常就跳过某个分支；更可靠的修法是回到状态机边界，确认哪个路径创建了临时占位，哪个路径应该释放它，再补一个只覆盖这个状态迁移的回归测试。

Agent 框架里也常见类似问题。一次会话恢复时，历史 tool response 没有重新 seed 到当前上下文，最后表现可能是 planner 重复调用工具。真正的修复不是在输出层“去重”，而是补齐 resume 时的状态恢复逻辑，并验证普通新会话不受影响。

## 改动要尊重仓库已有抽象

一个陌生仓库里，最诱人的做法是自己加一套 helper，把问题包起来。但高质量项目通常已经有自己的边界：错误类型、配置解析、事件循环、测试 fixture、序列化层、权限模型、缓存策略。一个 PR 如果绕过这些抽象，即使能工作，也会显得“不像这个项目的人写的”。

我更喜欢先找已有模式：相邻文件怎么处理同类错误，已有测试怎么构造 fixture，维护者之前 merge 过什么风格的修复。能复用本地约定，就少引入新概念。一个小 patch 如果正好贴合项目结构，review 成本会低很多。

这个细节很重要。成熟项目里，“能跑”和“应该这样放”不是一回事。SDK 项目通常已经有自己的 schema 层、transport 层和 retry 层；eval 项目会有 fixture、mock model、sandbox 或 release artifact 的约定；前端工具链会有 workspace、package boundary 和 lockfile 规则。把修复放错层，即使测试能过，也会让维护者担心后续维护成本。

## 验证要让 reviewer 省时间

PR 描述里最有价值的不是“我修了这个问题”，而是告诉 reviewer：怎么确认这个问题存在，怎么确认这个修复有效，哪些测试跑了，哪些测试因为环境限制没跑，失败是否和本补丁有关。

维护者没有义务替你补证据。如果一个 PR 只写 “fixed bug”，reviewer 要自己猜问题、猜边界、猜风险。反过来，如果描述里有复现、修复逻辑和 focused test，maintainer 能很快判断这个 PR 是否值得继续看。

我现在更偏向在 PR 描述里写很具体的验证边界，而不是写漂亮话。例如：

- 跑了哪个 focused test，为什么它覆盖这个 bug。
- 没跑全量测试，是因为本地缺 GPU、S3 artifact、workflow approval，还是因为全量测试成本过高。
- 如果 CI 红了，红项是当前 diff 引入，还是外部 gate / 平台权限 / release artifact。
- 如果修复涉及兼容性，旧行为和新行为分别是什么。

这些信息不会让 patch 自动变好，但能显著降低 reviewer 的不确定性。

## 不要为了数量开 PR

AI 工具让“多开几个 PR”变得很容易，但开源社区不是刷题平台。重复 PR、浅层 PR、没复现的 PR，会消耗维护者耐心，也会伤害自己的可信度。真正长期有价值的贡献，是你在某个项目里逐渐建立“这个人提交的 patch 通常靠谱”的印象。

我现在判断一个修复值不值得做，会先问几个问题：

- 这个问题是否真实存在于当前代码？
- 修复是否解决根因，而不是掩盖症状？
- diff 是否足够聚焦？
- 是否有 focused test 或可复现验证？
- 有没有重复 PR 或维护者已经选定的方向？
- 这个 PR 对项目和自己的技术展示是否都成立？

如果这些问题答不上来，宁愿不做。

我还会加一个更现实的检查：这个项目里我自己的 open PR 是否已经太多。如果一个仓库已经有十几条等待 review 的 PR，新开一个中等价值 PR 很可能不是加分，而是在增加队列噪声。更好的做法是先把旧 PR 里已经失效、被替代、方向不清的关掉；有价值但缺关注的，再用新的证据去 ping。

这也是我对 AI 编程工具的一个判断：它们最适合加速已经清楚的问题，而不是替代工程判断。AI 可以帮你读代码、补测试、生成候选 patch，但它不能替你确认问题是否仍然存在，也不能替你承担维护者视角下的可信度。

高质量开源贡献最终拼的不是手速，而是判断力。

如果用一句话总结：真正值得发的 PR，应该让维护者觉得“这个人减少了我的工作量”，而不是“这个人又给我制造了一个需要判断的 diff”。
